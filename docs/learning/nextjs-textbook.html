Next.js 実践教科書

既存の apps/web を土台に、web 版「メモ機能」を新しいページとして章ごとに積み上げながら App Router のベストプラクティスを体得する build-along 教材。

この本のねらい

参考書で一度読んだ知識を、実際に動くコードを自分の手で足していくことで「使える」レベルに引き上げます。各章は「概念 → 作るコード → 逐行解説 → 図解 → なぜこう書くか → ハンズオン＆クイズ」で構成しています。

前提と進め方

  • 対象: Next.js を書籍レベルで一度学んだ人。React の基本（JSX / props / hooks）は既知とします。
  • 教材の題材: この monorepo には memo の API（CRUD 完備）・スキーマ・mobile 実装が既にありますが、web にはまだメモ機能がありません。これを一から作ります。
  • この本の約束ごと: 既存 web のコードは改修しません。あなたが apps/web/src に新しいファイルを足していくのが学習の本体です。この HTML はその手引きです。
  • 動かし方: pnpm dev で web を起動し、ログインは /dev/login?as=alice（開発専用）を使います。詳しくは第0章。
📌 バージョンについて

本書は next@16 / react@19 を前提にしています。Next.js はキャッシュや API のデフォルトがメジャーバージョンで変わるため、細部は必ず公式ドキュメントの当該バージョンで確認してください。本文でも版依存の箇所には「互換メモ」を添えています。

0. 全体像と土台を知る

コードを足す前に、すでに用意されている土台を把握します。ここを理解しておくと、以降の章で「なぜ自分でゼロから認証や fetch ラッパを書かないのか」が腑に落ちます。

この章のゴール
  • apps/web の現在の構成と、これから作る memo 機能の位置づけを理解する
  • ディレクトリ規約（app → components → features の依存方向）を頭に入れる
  • 開発サーバの起動と dev ログインができる

0.1 いま web にあるもの

現状の apps/web はユーザー情報を表示するサンプル1枚と、認証まわりの土台だけがあります。

apps/web/src/ ├─ app/ │ ├─ layout.tsx # ルートレイアウト（フォント・metadata） │ ├─ page.tsx # トップ：ユーザー情報サンプル（Server Component） │ ├─ error.tsx # エラー境界 │ ├─ global-error.tsx # ルートレイアウトごと落ちた時の砦 │ ├─ not-found.tsx # 404 │ ├─ sign-in/ # サインイン画面 + Server Action │ ├─ dev/login/route.ts # 開発専用ログイン（Route Handler） │ └─ api/auth/callback/google/route.ts ├─ components/user/ # RefreshButton / Lottie（Client Component） ├─ features/user/get-user.ts # server-only なデータ取得 ├─ libs/ │ ├─ api-client.ts # ★ 認証付き fetch ラッパ（401→refresh 付き） │ ├─ auth.ts # cookie 読み書き │ └─ auth-actions.ts # logout の Server Action ├─ env.ts # Zod で環境変数を検証（server-only） └─ proxy.ts # middleware（Next.js 16 の新名称）

この章以降で 緑色の新規ファイルを足していき、最終的にこうなります。

apps/web/src/ ├─ app/memos/ │ ├─ page.tsx # 一覧（第1〜3章） │ ├─ loading.tsx # ローディングUI（第1章） │ ├─ error.tsx # セグメントのエラー境界（第9章） │ ├─ actions.ts # 作成/更新/削除の Server Action（第5〜6章） │ ├─ new/page.tsx # 作成フォーム（第5章） │ └─ [id]/ # 詳細・編集（第4・6章） ├─ components/features/memo/ # メモUIコンポーネント群（第3章〜） ├─ features/memo/ │ ├─ memo.api.ts # サーバ経由のAPI通信＋Zod検証（第2章） │ └─ memo.state.ts # Zustand（第7章） └─ app/api/memos/route.ts # クライアント動的取得用（第7章）

0.2 土台①：認証付き API クライアント

web の鉄則は「ブラウザから Express API を直接叩かない」。必ずサーバー側（Server Component / Server Action / Route Handler）を経由します。その心臓が libs/api-client.ts です。

なぜブラウザから直接 API を叩かないのか

①認証トークンを HttpOnly cookie に隠せる — JS から読めない cookie に置くことで XSS でトークンを盗まれにくくします。ブラウザ fetch では Authorization ヘッダを JS が組み立てる必要があり、トークンが JS に露出します。
②401→refresh の自動リトライを1か所に集約できる。
③ API の origin をサーバー側だけで解決でき、CORS も不要。
あなたが memo 機能で書く memo.api.ts も、この apiClient を呼ぶだけにします。

0.3 土台②：型はすべて @repo/api-schema から

リクエスト/レスポンスの型は web で独自定義しません。共有パッケージ @repo/api-schema の Zod スキーマと z.infer 型を import します。memo の型はすでに定義済みです。

⚠ mobile 実装をそのまま真似しない

apps/mobile/src/features/memo/memo.api.ts は axios でブラウザから直接 API を叩き、コメントにも「型安全ではない」と書かれています。これはアンチパターンの実例として残っているものです。web ではこの章で学ぶベストプラクティス版（サーバ経由 + スキーマ検証）を作ります。「既存コードだから正しい」とは限らない、という視点も持ってください。

0.4 依存の方向：app → components → features

このプロジェクトはフロント3アプリ（web / admin / mobile）でディレクトリ規約を統一しています。技術単位（hooks/ や utils/）ではなく機能単位で分け、依存の向きを一方向に保ちます。

ディレクトリ役割memo での例
app/memos/ルーティング＋ページ。薄く保つ（コンポーネントを組み合わせるだけ）page.tsx, [id]/page.tsx, actions.ts
components/features/memo/memo 固有の UI コンポーネントMemoList.tsx, MemoForm.tsx
components/ui/汎用 UI（Button, Input …）Button.tsx
features/memo/ロジックのみ（レンダリングなし）。API 通信・状態管理memo.api.ts, memo.state.ts

0.5 開発サーバを起動してログインする

なぜ dev ログインがあるのか

本番の Google OAuth を毎回踏むのは開発上つらいので、NODE_ENV !== "production" のときだけ有効な近道が用意されています。proxy.ts・Route Handler・API の3か所でガードされ、本番では 404 になります（第8章で詳解）。

🛠 ハンズオン 0
  1. pnpm --filter web dev で web を起動する。
  2. http://localhost:3000 を開く → 未ログインなら /sign-in にリダイレクトされることを確認。
  3. http://localhost:3000/dev/login?as=alice を踏み、トップでユーザー情報が表示されることを確認。
  4. apps/web/src/libs/api-client.ts を実際に開き、get/post/put/delete がどんな型を返すか眺めておく（第2章で使います）。
web では「ブラウザから Express API を直接 fetch する」のはなぜ禁止？ 主な理由を1つ挙げよ。
認証トークンを HttpOnly cookie に隠したまま、サーバー側でのみ Authorization ヘッダに載せられるから（XSS 対策）。加えて 401→refresh の集約、origin のサーバー解決、CORS 不要といった利点もある。トークンをブラウザ JS が扱うと露出リスクが上がる。

1. App Router のファイル規約

App Router ではファイル名そのものが役割を持ちます。page.tsx を置けばルートになり、loading.tsx を隣に置くだけでローディング UI が有効になる。まずは空の /memos ページを作りながら、この「規約」を体で覚えます。

この章のゴール
  • app/ 配下の特殊ファイル（page / layout / loading / error / not-found）の役割を説明できる
  • ルートセグメントとルートグループ (...) の違いを理解する
  • 動く /memos ページ（静的な仮表示）と loading.tsx を作る

1.1 特殊ファイルの地図

App Router で意味を持つ主なファイルは次の通り。1つのフォルダ＝1つの URL セグメントで、その中に役割別のファイルを置きます。

ファイル役割境界(Server/Client)
page.tsxその URL の本体。これがある階層だけが公開ルートになる既定 Server
layout.tsx配下で共有される枠。再ナビゲーションでも再マウントされない既定 Server
loading.tsx配下の読み込み中に自動表示（内部的に <Suspense>）Server/Client 可
error.tsx配下で throw された実行時エラーを捕捉する境界Client 必須
not-found.tsxnotFound() 呼び出し・未定義 URL の表示Server 可
route.tsページではなく HTTP エンドポイント（GET/POST…）。page.tsx と同居不可Server
template.tsxlayout に似るが、遷移ごとに再マウントする枠Server/Client 可
なぜ「ファイル名が規約」なのが嬉しいのか

ルーティング設定やローディング/エラーの配線をコードで書かなくてよい。フォルダを見れば URL 構造と、どこにローディング・エラー処理があるかが一目で分かります。「設定より規約(convention over configuration)」の思想です。

1.2 まず動く空ページを置く

app/memos/page.tsx を作れば、それだけで /memos が生えます。まずはデータ取得なしの静的な仮表示から。

逐行のポイント

  • export default したコンポーネントがそのままページになる。関数名は自由（URL には無関係）。
  • "use client" を書いていないのでServer Component。サーバーでレンダリングされ、JS バンドルに載らない。
  • export const metadata で <title> などを宣言（詳細は第4章の Metadata API）。
  • web / admin の規約ではコンポーネントは function 宣言で書きます（API 側はアロー関数）。page.tsx は薄く保つのが原則。

1.3 loading.tsx を隣に置く

同じ階層に loading.tsx を置くと、この配下が「まだ準備中」の間 Next.js が自動でそれを表示します（次章でデータ取得を入れると効いてきます）。裏側では Next.js がページを <Suspense fallback={<Loading/>}> で包んでくれています。

1.4 ルートグループ (...) — URL を増やさずに整理する

フォルダ名を ( ) で囲むと、その階層はURL に現れません。共通レイアウトを与えたいページ群をまとめるのに使います。例えばログイン後の画面をまとめた (dashboard) グループ。

app/ ├─ (dashboard)/ # ← URL には出ない。ダッシュボード共通レイアウト用 │ ├─ layout.tsx # サイドバー等の共通枠 │ ├─ memos/page.tsx # → /memos │ └─ settings/page.tsx # → /settings └─ sign-in/page.tsx # → /sign-in（レイアウトを共有しない）
本書での扱い

本書では話を単純にするため app/memos/ 直下に作ります。実務でヘッダ付きの共通レイアウトを与えたくなったら app/(dashboard)/memos/ に移し、(dashboard)/layout.tsx を足す、という発展を第3章のコラムで触れます。CLAUDE.md でも Server Action の配置例として app/(dashboard)/categories/actions.ts が示されています。

🔧 互換メモ：params / searchParams は Promise

Next.js 15 以降、page.tsx が受け取る params と searchParams は Promise になりました（await が必要）。既存の sign-in/page.tsx でも const { error } = await searchParams と書かれています。第4・5章で実際に使います。

🛠 ハンズオン 1
  1. apps/web/src/app/memos/page.tsx を上記の内容で新規作成する。
  2. /memos にアクセスし、見出しが表示されることを確認（未ログインなら先に /dev/login?as=alice）。
  3. loading.tsx も作成する。今はまだデータ取得が無いので一瞬しか出ないが、第2章で効いてくる。
  4. 試しに page.tsx の関数名を Foo に変えても /memos が動くことを確認 → 関数名は URL に無関係だと体感する。
layout.tsx と template.tsx の一番の違いは？
再マウントされるか否か。layout はページ間遷移でも状態を保持したまま再マウントされない（共有の枠として残る）。template は遷移ごとに新しいインスタンスとして再マウントされるので、遷移のたびに入場アニメを走らせたい／useEffect を毎回動かしたい、といった時に使う。
error.tsx だけが "use client" 必須なのはなぜ？
エラー境界は React の Error Boundary の仕組みを使い、reset() で再試行するためにクライアント側の状態とイベントを扱う必要があるから。Server Component は状態やイベントを持てないため、境界はクライアントで動く必要がある。

2. データ取得 — Server Component で一覧を出す

いよいよメモ一覧を API から取ってきます。ここが App Router の一番おいしいところ：Server Component の中で直接 await して fetch できる。useEffect + useState のローディング地獄が消えます。

この章のゴール
  • Server Component で async/await データ取得ができる
  • server-only と @repo/api-schema による検証で「型安全なデータ取得層」を作る
  • キャッシュと動的レンダリングの判断基準（cookies() を使うと動的になる 等）を理解する

2.1 データ取得層 memo.api.ts を作る

規約どおり、API 通信ロジックは features/memo/memo.api.ts（ロジックのみ・UI 非依存）に置きます。土台の apiClient を呼び、返り値をスキーマで検証してから返すのがポイント。

逐行のポイント

  • import "server-only"：この関数が誤ってクライアントに混入したらビルドで落とすお守り。トークンやサーバ処理が漏れない。
  • apiClient.get<unknown>：apiClient の <T> はただの型キャストで実行時検証はしない。だから unknown で受けて…
  • …getMemoListResponseSchema.parse(raw) で実行時に形を保証する。API が予期せぬ形を返したらここで例外になり、壊れたデータが画面まで到達しない。
なぜ as ではなく .parse() なのか

mobile の memo.api.ts は new Memo(result.data) とキャストするだけで、コメントに「型安全ではない」と正直に書かれています。TypeScript のキャストは「コンパイラを黙らせる」だけで、実行時には何も守りません。API のバージョン差やバグで title が欠けていても、キャストは素通りして UI で undefined が表示される。@repo/api-schema の .parse() を通せば、契約と実データのズレを境界で検出できます。これが「スキーマをスタック横断で共有する」設計の狙いです。

2.2 ページから呼ぶ

page.tsx を Server Component のまま async にして、memoApi.getList() を await します。第1章の静的ページを置き換えます。

2.3 キャッシュと動的レンダリング

「このページ、いつのデータが出るの？」を決めるのがキャッシュ／レンダリング戦略です。Next.js 15 以降のデフォルトを押さえましょう。

やりたいこと書き方結果
毎回最新（動的）既定。cookies()/headers() を使うと自動でこうなるリクエストごとにサーバー実行
N 秒キャッシュexport const revalidate = 6060秒ごとに裏で再生成（ISR）
常に動的を明示export const dynamic = "force-dynamic"キャッシュ無効
fetch 単位で制御fetch(url, { next: { revalidate: 60 } })その取得だけキャッシュ
🔧 互換メモ：本書のメモ一覧は「動的」

本書の apiClient は内部で cookies()（next/headers）を読んでトークンを付けます。cookies() を使った時点でそのルートは動的レンダリングになるため、メモ一覧は常にリクエスト時取得です。ログイン中ユーザー固有のデータなのでこれで正しい。逆に「全員同じ・更新が稀」なデータ（公開記事一覧など）は revalidate でキャッシュした方が速くなります。
なお Next.js 15 以降、fetch はデフォルトでキャッシュされません（14 までとは逆）。キャッシュしたい時だけ明示的にオプトインします。

エラーはどこで拾う？

page.tsx では try/catch せず、失敗時は例外を投げっぱなしにしています。これは第9章で作る app/memos/error.tsx（セグメントのエラー境界）に自動で捕捉させる方針だからです。「正常系はページ、異常系は境界」と役割を分けると、ページ本体が読みやすく保てます。トップの page.tsx が try/catch しているのは、あえて「その場でエラー表示する」別方針の例です。

🛠 ハンズオン 2
  1. features/memo/memo.api.ts を作成する。
  2. app/memos/page.tsx を上記の取得版に更新する。
  3. API とスキーマを起動：pnpm --filter @repo/api-schema build → API も pnpm --filter api dev（別ターミナル）。
  4. /memos を開き、一覧が出ることを確認（データが無ければ「まだメモがありません」）。
  5. 実験：memo.api.ts の .parse(raw) を一時的に raw as GetMemoListResponse に変え、API を止めて壊れたレスポンスを返すと何が起きるか観察 → .parse に戻す。
Server Component でのデータ取得が、Client Component の useEffect+fetch より優れている点を2つ。
① ウォーターフォールとローディング状態が消える：サーバーで await して完成 HTML を返すので、クライアントで「読み込み中→データ」の二段階レンダリングが不要。② 秘密を隠せる：API トークンや API の URL、重いロジックがクライアント JS に載らない。加えて DB/API に物理的に近いサーバーで叩ける、バンドルが軽くなる、等。
このメモ一覧ページが「動的レンダリング」になる決め手は？
apiClient が内部で cookies() を呼んでいること。リクエスト固有の情報（cookie）に依存するルートは、Next.js が自動的に動的（リクエストごとに実行）と判定する。ビルド時に固定 HTML を作れないため。

3. コンポーネント設計 — UI を正しく分割する

第2章の page.tsx は取得もマークアップも全部入りでした。実務ではすぐ肥大化します。ここで規約どおり UI を components/ に切り出し、page.tsx を薄くします。「どこに置くか」「Server と Client のどちらにするか」を判断できるようになるのがゴール。

この章のゴール
  • app → components → features の依存方向に沿って UI を分割できる
  • コンポーネント単位で Server / Client を判断できる（デフォルトは Server、必要な島だけ Client）
  • 「Server Component を Client Component の children に渡す」合成パターンを使える

3.1 分割方針

一覧画面を3つに割ります。ロジックは features/、見た目は components/、束ねるのは app/。

app/memos/page.tsx # 取得して components に渡すだけ（薄い） components/features/memo/ ├─ MemoList.tsx # memos[] を受け取り並べる（Server） ├─ MemoListItem.tsx # 1件の見た目（Server） └─ MemoEmptyState.tsx # 0件時の表示（Server）

MemoListItem — 1件分の見た目

MemoList / MemoEmptyState

page.tsx は薄くなる

なぜ page.tsx を薄く保つのか

page.tsx は「このURLで何を取得し、どの部品を組むか」だけを語る司令塔にします。見た目の詳細（クラス名やマークアップ）はコンポーネント側へ。こうすると ①ページを読むだけで画面構成が分かる、②同じ MemoList を検索結果やダッシュボードで再利用できる、③見た目の変更が page.tsx に波及しない、という利点が出ます。規約の「app/ は薄く」はこの狙いです。

3.2 Server か Client か — 島（island）で考える

App Router ではすべてデフォルト Server Component。"use client" を書いたファイルとそこから import される先だけがクライアントに送られます。だから設計の合言葉は「できるだけ Server、インタラクションが要る所だけ Client の小島にする」。

上の MemoList / MemoListItem / MemoEmptyState は表示だけなので全部 Server。onClick や useState が要る部品（例：削除ボタン、検索ボックス）だけを "use client" の小島に切り出します。既存の refresh-button.tsx や Lottie の user-bot-animation.tsx がまさにその実例です。

Client Component が要るサイン例
onClick / onChange などのイベント削除ボタン、フォーム操作
useState / useEffect / useRef開閉トグル、入力状態
ブラウザ API（window, localStorage）テーマ切替
状態管理ライブラリ（Zustand 等）クライアント状態（第7章）
DOM 依存ライブラリLottie（dynamic(ssr:false)）
⚠ よくある誤解："use client" は「ページ全体」に付けない

「フォームがあるからページ全部 Client」は最悪手。ページを Client にすると配下の Server の利点（サーバー取得・軽いバンドル）を全部失います。正解は「ページは Server のまま、ボタン1個だけ Client の小島」。境界はできるだけ葉（末端）に押し下げる。

3.3 合成パターン：Server を Client の children に渡す

「Client Component の中で Server Component を使いたい」ことがあります（例：開閉できるパネルの中にサーバー取得の中身を入れる）。Client の中で Server を import することはできませんが、children として渡すのは OK です。

「Client Component の中では Server Component を import できない」のに、children で渡せるのはなぜ？
Server Component のレンダリング（サーバーでの実行）は親（Server）側で先に済んで、その結果が「描画済みの子」として Client に渡るから。Client が import して自分でレンダリングしようとすると、サーバー専用処理をクライアントで走らせることになり成立しない。children なら Client は「出来上がった中身をどこに置くか」を決めるだけで、中身を実行しない。
Server/Client の境界は「根（ページ）」と「葉（末端）」どちらに寄せるべき？なぜ？
葉に寄せる。"use client" は付けたファイルとその配下すべてをクライアントに送る。根に付けると木全体が Client 化してサーバーの恩恵を失う。インタラクションが必要な最小の部品にだけ付けるのがベストプラクティス。

4. 動的ルートと詳細ページ

一覧の各項目は /memos/1, /memos/2 … と ID で開けるようにします。URL の一部を変数として受け取る動的ルート [id] を作りながら、await params・notFound()・generateMetadata を学びます。

この章のゴール

  • 存在しない ID で notFound() → not-found.tsx に繋げられる
  • generateMetadata でページごとの <title> を動的生成できる
  • ルートパラメータも Zod で検証する（プロジェクト規約）

4.2 [id] ページを作る

フォルダ名を [id] にすると、その部分の URL が params.id として渡ります。Next.js 15 以降、params は Promise なので await します。

逐行のポイント

  • getMemoPathParamSchema は z.coerce.number().int().positive()。文字列→数値変換と検証を Zod に任せる（Number() や parseInt を手書きしない、が規約）。
  • 不正 ID（/memos/abc）は safeParse 失敗 → notFound()。存在しない ID は API が 404 → ApiClientError.status === 404 を見て notFound()。
  • notFound() は最寄りの not-found.tsx を表示する。既存の app/not-found.tsx がそのまま使われる（第9章でメモ専用に上書きも可）。
  • 404 以外の失敗（500 等）は throw して error.tsx に任せる。「無い」と「壊れた」を区別するのが ApiClientError が status を保持している狙い（第9章）。
なぜ generateMetadata をわざわざ書くのか

静的な export const metadata ではページ内容に応じた <title> を作れません。generateMetadata は params を受け取れる非同期関数なので、「メモのタイトルをそのままページタイトルに」といった動的な SEO / タブ表示ができます。Next.js は本文レンダリングと generateMetadata の getDetail を重複取得しないよう自動でメモ化（dedupe）してくれるので、2回書いても API は1回で済みます。

4.3 SSG / ISR はこのページに使う？

generateStaticParams を書くと、ビルド時に [id] の一覧を先に生成して静的化（SSG）できます。ブログ記事のような「全員に同じ・更新が稀」なコンテンツでは強力です。

メモは動的でよい

メモはログインユーザー固有・頻繁に増減し、apiClient が cookies() を使う（＝動的）ため、SSG/ISR は不向きです。「認証が要る・ユーザーごとに違う・すぐ変わる」データは動的レンダリングが正解。逆に公開・共通・低頻度更新なら ISR を検討する、という判断軸を持っておきましょう。

🔧 互換メモ：動的 API はすべて Promise

Next.js 15/16 では params / searchParams / cookies() / headers() がすべて非同期（Promise）です。同期的に params.id と書くと型エラー・実行時警告になります。必ず await しましょう。

5. Server Actions — 作成フォーム

ここからはデータ変更（mutation）。App Router では、フォーム送信を Server Action（"use server" 関数）で受けます。API ルートを別途作らなくても、関数を form の action に渡すだけでサーバー処理が呼べます。既存の startGoogleOAuth / logoutAction と同じ仕組みです。

この章のゴール
  • Server Action を書き、form action から呼べる
  • useActionState でバリデーションエラーを画面に戻せる
  • useFormStatus で送信中の UI を出せる
  • revalidatePath + redirect で「作成→一覧へ・最新表示」を実現できる

5.1 取得層に create を足す

5.2 Server Action を書く（actions.ts）

規約では Server Action は対応するページと同じディレクトリに actions.ts として置きます（app/actions/ のような共通置き場は作らない）。

⚠ redirect() は try/catch の外へ

redirect() は内部的に特別な例外（NEXT_REDIRECT）を throw して遷移を実現します。try/catch の中に置くと、その catch が遷移用の例外を握りつぶしてリダイレクトが効きません。必ず try ブロックの外で呼びます。revalidatePath も同様に成功後に。

5.3 フォーム UI（Client の島）

入力とエラー表示はインタラクティブなので Client Component にします。useActionState が [state, formAction, isPending] を返し、state にはさっきの CreateMemoState（＝エラー）が入ります。

作成ページ（Server）

なぜ検証を「サーバー側でも」やるのか

フォームの入力チェックは HTML の required やクライアント JS でもできますが、それらは簡単に回避されます（開発者ツール、直接 POST）。サーバーに届いた入力は必ずサーバーで検証するのが鉄則。しかも createMemoRequestSchema は API 側と同じスキーマなので、フロントと API で検証ルールがズレません。これが「スキーマをスタック横断で共有」する実利です。

Server Action と Route Handler、どちらを使う？

プロジェクト規約：データ変更（POST/PUT/DELETE）は基本 Server Action、データ取得（GET）は Server Component か Route Handler。「Server Action を GET 相当に使ってはいけない（mutation 専用）」。ファイルアップロードや外部公開 API など Server Action が苦手なケースだけ Route Handler にします（作成フォームは Server Action が最適）。

7. 状態管理 — Zustand と「サーバ状態 vs クライアント状態」

「状態管理ライブラリを入れる前に、そもそも何をクライアントで持つべきか」を先に決めるのがこの章の肝です。App Router ではサーバーで取れるデータはサーバーに任せ、Zustand は本当にクライアント固有の UI 状態にだけ使います。題材として「メモの絞り込み検索」を作ります。

この章のゴール
  • サーバ状態とクライアント状態を切り分けられる（これが最重要）
  • Client 動的取得のための Route Handler を作れる
  • Zustand ストアを規約（アクション名に Store）どおりに書ける

7.1 まず切り分ける：その状態、どこが持ち主？

⚠ アンチパターン：サーバ取得したデータを Zustand に詰める

「メモ一覧を useMemoStore に入れて全画面で共有」…は避けます。サーバ状態を手動でクライアントに複製すると、再検証（revalidate）と二重管理になり「作成したのにストアが古い」バグの温床に。サーバ状態の単一の真実はサーバー。Zustand が持つのは「検索キーワード」「選択中の ID」「サイドバー開閉」のようなUI の気分だけにします。
参考：mobile はクライアント主体アプリなので useMemoStore にメモ配列を持ちますが、web（Server Component が使える）では方針が異なります。環境に応じて最適解は変わるという良い対比です。

7.2 Client 動的取得用の Route Handler

検索はキー入力のたびにサーバーへ問い合わせます。これは「Client Component からの動的取得」なので、規約どおり Route Handler（app/api/*/route.ts）を作り、そこへ同一オリジンで fetch します（ブラウザから Express を直接叩かない原則は維持）。

なぜ検索は Server Action ではなく Route Handler？

検索はデータ取得（GET 相当）だから。規約で「Server Action は mutation 専用、GET に使ってはいけない」と定めています。取得は Server Component（初期表示）か Route Handler（クライアントからの動的取得）に振り分けます。今回はキー入力に応じた動的取得なので Route Handler が正解。

7.3 Zustand ストア（規約：アクション名に Store）

検索の キーワード・結果・ローディングを Zustand で持ちます。プロジェクト規約でアクション関数名には必ず Store を含めます（setStoreQuery など）。これは API 呼び出しやハンドラの setQuery 等と名前が衝突して再帰バグを起こすのを防ぐためです。

7.4 検索 UI（Client 島）

なぜ useMemoStore((s) => s.query) と1つずつ選ぶのか

Zustand はセレクタで購読を細かく分けると、その値が変わった時だけ再レンダリングされます。const { query, results } = useMemoStore() のように丸ごと取ると、ストアのどれか1つでも変わるたびに再描画されて無駄が出ます。パフォーマンス上はセレクタ分割が基本です。

URL に載せる選択肢（searchParams）

検索語を「共有・ブックマーク・戻る操作」で復元したいなら、Zustand ではなく URL の searchParams（?q=...）に載せ、Server Component で await searchParams して初期描画する手もあります。「共有したい状態は URL、共有したくない一時的な UI 状態は Zustand/useState」という判断軸を持つと設計がぶれません。

8. ミドルウェアと認証

ここまで作った /memos 系ページは、実はすでに認証で守られています。その仕組みが proxy.ts（＝ middleware）と cookie 設計、そして apiClient の 401→refresh です。自分で認証コードを書かずに済んでいる理由を、土台のコードを読み解いて理解します。

この章のゴール
  • proxy.ts が「入口の門番」として何をしているか説明できる
  • HttpOnly / SameSite=lax cookie の設計理由を理解する
  • OAuth ログインの一連の流れを追える
  • Edge ランタイムの制約を知る
🔧 互換メモ：middleware.ts → proxy.ts

Next.js 16 で従来の middleware.ts は proxy.ts に改名されました（本プロジェクトは src/proxy.ts が proxy 関数と config を export）。役割は同じ「全リクエストの前段で走る関所」です。旧バージョンの資料では middleware という名前で説明されている点に注意。

8.1 門番 proxy.ts を読む

逐行のポイント

  • すべてのリクエストの前に走る（config.matcher で対象を絞る）。だから /memos に個別のガードを書かなくても守られる。
  • 公開したいパスは PUBLIC_PATHS に列挙。matchesPathPrefix でセグメント境界一致させ、/sign-in が /sign-in-foo まで通す事故を防ぐ。
  • JWT の中身は検証しない。cookie が「有るか無いか」だけ見る（理由は次項）。
  • hasRefresh だけでも通す：access が切れていても、ページ側の apiClient が 401→refresh で復活させる機会を残すため。
なぜ proxy で JWT を「検証しない」のか

proxy.ts は Edge ランタイムで動きます。Edge は軽量で高速ですが、Node.js の暗号ライブラリ等がフルには使えず、署名検証のような重い処理に不向きです。そこで proxy は「cookie の有無」で粗くふるい分けるだけにして、本物の JWT 検証は API 側（Node ランタイム）に任せます。「関所は軽く、厳密な認可はデータに近い層で」という多層防御の考え方です。

8.2 cookie 設計

なぜ sameSite: "lax" なのか

OAuth は「Google（外部サイト）→ 自サイトのコールバック」というクロスサイト起点の遷移です。strict にすると、この最終遷移で cookie が送られず「ログインしたのに未認証扱い」になります。lax はトップレベルのページ遷移では cookie を送るのでこれを回避。CSRF 対策は別途 state パラメータ（startGoogleOAuth で生成）で担保しています。短命な access（15分）＋長命な refresh（7日）の二段構えも典型パターンです。

8.3 OAuth ログインの全体像

8.4 401→refresh の自動リカバリ（再掲）

access（15分）が切れると API は 401 を返します。apiClient はそれを検知して refresh トークンで更新し、一度だけ元のリクエストを再試行します（第0章のコード）。だから proxy は access が切れていても hasRefresh で入場を許可するのでした。この一連が「ユーザーは 15 分ごとに再ログインさせられない」体験を支えています。

🛠 ハンズオン 8
  1. ログアウト状態で /memos にアクセス → /sign-in?redirect=/memos に飛ばされることを確認（proxy の働き）。
  2. ブラウザの開発者ツール → Application → Cookies で app_access_token / app_refresh_token が HttpOnly になっていることを確認。
  3. app_access_token だけ削除して /memos を再読込 → それでも表示される（refresh が効く）ことを確認。
  4. proxy.ts の PUBLIC_PATHS を眺め、なぜ /api/auth/callback/google が公開なのか説明してみる。
proxy.ts が「cookie の有無だけ」を見て中身を検証しないのは、どんな制約と設計判断から？
proxy は Edge ランタイムで動き、JWT 署名検証のような処理に不向き・重い。だから関所は軽く（有無で粗くふるい分け）、厳密な検証は API（Node ランタイム）に集約する多層防御にしている。access 切れでも refresh があれば通すのも、ページ側の 401→refresh に復活の機会を残すため。

9. エラーハンドリングと堅牢化

正常系ができたので、失敗した時に何が起きるかを設計します。App Router は error.tsx / not-found.tsx / loading.tsx という境界ファイルで、宣言的にエラーとローディングを配線できます。ここではメモ専用の境界を足し、ApiClientError で「無い」と「壊れた」を区別し、Suspense で段階表示します。

この章のゴール
  • セグメント単位の error.tsx を書ける
  • ApiClientError.status で 4xx/5xx を出し分けられる
  • <Suspense> で部分ストリーミングできる

9.1 エラーの3つの砦

ファイル捕まえる対象どこに置く
error.tsx配下の Server/Client で throw された実行時エラーセグメントごと（例 app/memos/）
not-found.tsxnotFound() 呼び出し・未定義 URLセグメントごと / ルート
global-error.tsxルートレイアウト自体が落ちた最悪ケースルートのみ（既存）

error.tsx は自分と同じ階層の layout は捕捉できない（レイアウトはエラー境界の外側）点に注意。/memos 配下の throw は app/memos/error.tsx があればそこで、無ければ上位の app/error.tsx（既存）で捕まります。

9.2 メモ専用のエラー境界

第2章で page.tsx は失敗時に throw する方針にしました。その受け皿を作ります。

なぜ「正常系はページ、異常系は境界」に分けるのか

page.tsx の中を try/catch だらけにすると、本来の表示ロジックがエラー処理に埋もれます。App Router は「throw したら最寄りの error.tsx が受ける」という宣言的な仕組みを持つので、ページは正常系だけを書き、失敗は境界に委譲できます。reset() は「同じセグメントをもう一度レンダリングし直す」再試行ボタンに使えます。digest はサーバーで詳細ログに紐づく ID で、ユーザーには詳細を隠しつつ調査可能にします。

9.3 「無い」と「壊れた」を分ける — ApiClientError

土台の apiClient は non-2xx を ApiClientError(status, body) として throw します。status を保持しているおかげで、呼び出し側が 404（無い）と 500（壊れた）を区別できます。第4章の詳細ページはこれを使って「404 は notFound()、それ以外は throw して error.tsx」と振り分けていました。

昔なぜ困ったか（コメントより）

以前は generic な Error を投げていて status が失われ、Route Handler が全部 500 に丸めていました。すると「入力ミス（4xx）」と「サーバー障害（5xx）」をフロントで区別できない。エラーに構造（status）を持たせると、UI 側で適切な出し分けができます。これは「エラーも型設計の対象」という良い教訓です。

9.4 Suspense で段階表示（部分ストリーミング）

loading.tsx は「ページ全体」の待ちを扱いました。ページの一部だけ遅い時は、その部分を <Suspense> で包むと、速い部分を先に見せて遅い部分だけ後から流し込む（ストリーミング）ことができます。

loading.tsx と <Suspense> の使い分け

loading.tsx ＝ ルート全体の Suspense（遷移時にページ丸ごとのフォールバック）。手書きの <Suspense> ＝ ページ内の一部分のフォールバック。ヘッダやナビはすぐ出し、重い一覧やレコメンドだけスケルトンにして後から流す、といった体感速度の作り込みができます。将来的な PPR（Partial Prerendering）はこの延長で、静的な殻を即返し動的部分を流す仕組みです（第10章）。

🛠 ハンズオン 9
  1. app/memos/error.tsx を作成する。
  2. わざと壊す：memo.api.ts の URL を /api/memo-broken に一時変更 → /memos でメモ用エラー画面＋「再試行」が出ることを確認 → 戻す。
  3. /memos/999999 で 404（notFound()）になり、500 のエラー画面ではないことを確認 → 「無い」と「壊れた」の区別を体感。
  4. （任意）一覧を <Suspense> + 切り出した MemoListAsync に変え、見出しが先に出ることを確認。
app/memos/error.tsx は、同じ階層の app/memos/layout.tsx で起きたエラーを捕捉できる？
できない。error.tsx はその内側（page や子）で throw されたものを捕まえる境界で、同階層の layout はその外側にあるため捕捉範囲外。layout のエラーは1つ上の階層の error.tsx（または global-error.tsx）が受ける。
詳細ページで 404 は notFound()、500 は throw、と分けられるのはなぜ？
apiClient が投げる ApiClientError が status を保持しているから。error.status === 404 なら「リソースが無い」＝ notFound()（not-found.tsx）、それ以外は「壊れた」＝ throw して error.tsx に回す、とステータスコードで意味づけできる。

10. 発展：パフォーマンスとテスト

最後に、作ったメモ機能を「速く・壊れにくく」する視点をまとめます。個々は深掘りせず、App Router で効く勘所とこのプロジェクトのテスト方針に絞ります。

この章のゴール
  • 画像・フォント・バンドルの基本最適化の勘所を持つ
  • キャッシュ / ストリーミング / PPR を「速さ」の観点で整理できる
  • このプロジェクトの UI 動作確認（Playwright MCP）の型を知る

10.1 パフォーマンスの勘所

App Router のパフォーマンス設計は、突き詰めると「クライアントに送る JS を減らす」ことです。"use client" を根に付けない、状態はできるだけサーバーに置く、島は葉に押し下げる——第3章・第7章でやったことが、そのままパフォーマンス最適化になっています。個別テクニックより境界設計が効きます。

🔧 互換メモ：PPR（Partial Prerendering）

静的な殻を即座に返し、動的部分（Suspense 内）をストリーミングで後追いする仕組み。Next.js の実験的機能として進化中で、バージョンにより有効化方法や安定度が異なります。採用する際は必ず使用中バージョンの公式ドキュメントで対応状況を確認してください。考え方は第9章のストリーミングの延長線上です。

10.2 テストと動作確認

このプロジェクトは UI を実装したら Playwright MCP で実画面を確認することを必須にしています（apps/web/CLAUDE.md）。pnpm build が通るのは型とルート登録の確認でしかなく、レンダリング不具合・hydration エラー・コンソールエラー・認証フローは検出できないためです。

最低限の確認項目（memo 機能に当てはめると）
  1. 着地 URL：/memos にアクセスして /memos のまま表示される（proxy に /sign-in へ飛ばされて OK 判定しない）。認証必須ページは pnpm --filter api issue-test-token <userId> で発行した JWT を app_access_token / app_refresh_token cookie に注入して検証する。
  2. コンソール：level: "error" が 0 件（hydration エラー等が無い）。
  3. 要素：一覧の見出し・各メモ・「新規作成」ボタン、フォームの各項目が表示される。
  4. フロー：作成 → 一覧に増える、編集 → 反映、削除 → 消える、を実操作で確認。

手順は verify-web-page skill にまとまっており、UI 実装直後に自発的に呼ぶ運用です。

Server Component / Server Action はどうテストする？

Server Component は「入力（props / 取得データ）→ 出力（要素）」の関数に近いので、取得層（memo.api.ts）を単体テストし、画面は Playwright の E2E で担保する二段構えが実務的です。Server Action も「Zod 検証 → API 呼び出し → revalidate/redirect」という純粋な処理単位なので、memoApi をモックして検証分岐（成功/バリデーションエラー/API失敗）を関数テストできます。UI と結合した振る舞いは E2E に任せる、と役割分担すると費用対効果が良いです。

🛠 ハンズオン 10（総仕上げ）
  1. ここまでの memo 機能を通しで動かし、作成→一覧→詳細→編集→削除→検索が繋がることを確認。
  2. ブラウザのコンソールに error が出ていないか確認（hydration エラーの有無）。
  3. 各ページの Client 島（"use client"）を数え、「本当に必要な最小限か」を見直す。
  4. （余力があれば）memo.api.ts の getList に対する簡単な関数テストを書いてみる。
pnpm build が成功したら「動作確認済み」と言ってよい？
ダメ。build は型チェックとルート登録の確認に過ぎず、実際のレンダリング崩れ・hydration エラー・コンソールエラー・認証リダイレクトの挙動は検出できない。実画面を（このプロジェクトでは Playwright MCP で）確認して初めて「動作確認済み」と言える。

Server / Client 判断フロー

  1. まずすべて Server Componentで書き始める（"use client" を書かない）。
  2. 次のどれかが要るか？ → onClick/onChange、useState/useEffect/useRef、window/localStorage、Zustand、DOM 依存ライブラリ。
  3. 要るなら、その部品だけを "use client" の小島に切り出す（ページ全体には付けない）。
  4. Client の中で Server の中身を出したい → import せず children で渡す。
  5. データは？ → 初期表示は Server Component で取得、mutation は Server Action、クライアント動的取得は Route Handler。
  6. 状態は？ → サーバ状態はサーバーに任せる。クライアント状態は useState → (共有なら) Zustand、(URL共有なら) searchParams。
