2. データ取得 — Server Component で一覧を出す

いよいよメモ一覧を API から取ってきます。ここが App Router の一番おいしいところ：Server Component の中で直接 await して fetch できる。useEffect + useState のローディング地獄が消えます。

この章のゴール
  • Server Component で async/await データ取得ができる
  • server-only と @repo/api-schema による検証で「型安全なデータ取得層」を作る
  • キャッシュと動的レンダリングの判断基準（cookies() を使うと動的になる 等）を理解する

2.1 データ取得層 memo.api.ts を作る

規約どおり、API 通信ロジックは features/memo/memo.api.ts（ロジックのみ・UI 非依存）に置きます。土台の apiClient を呼び、返り値をスキーマで検証してから返すのがポイント。

逐行のポイント

  • import "server-only"：この関数が誤ってクライアントに混入したらビルドで落とすお守り。トークンやサーバ処理が漏れない。
  • apiClient.get<unknown>：apiClient の <T> はただの型キャストで実行時検証はしない。だから unknown で受けて…
  • …getMemoListResponseSchema.parse(raw) で実行時に形を保証する。API が予期せぬ形を返したらここで例外になり、壊れたデータが画面まで到達しない。
なぜ as ではなく .parse() なのか

mobile の memo.api.ts は new Memo(result.data) とキャストするだけで、コメントに「型安全ではない」と正直に書かれています。TypeScript のキャストは「コンパイラを黙らせる」だけで、実行時には何も守りません。API のバージョン差やバグで title が欠けていても、キャストは素通りして UI で undefined が表示される。@repo/api-schema の .parse() を通せば、契約と実データのズレを境界で検出できます。これが「スキーマをスタック横断で共有する」設計の狙いです。

2.3 キャッシュと動的レンダリング

「このページ、いつのデータが出るの？」を決めるのがキャッシュ／レンダリング戦略です。Next.js 15 以降のデフォルトを押さえましょう。

やりたいこと書き方結果
毎回最新（動的）既定。cookies()/headers() を使うと自動でこうなるリクエストごとにサーバー実行
N 秒キャッシュexport const revalidate = 6060秒ごとに裏で再生成（ISR）
常に動的を明示export const dynamic = "force-dynamic"キャッシュ無効
fetch 単位で制御fetch(url, { next: { revalidate: 60 } })その取得だけキャッシュ
🔧 互換メモ：本書のメモ一覧は「動的」

本書の apiClient は内部で cookies()（next/headers）を読んでトークンを付けます。cookies() を使った時点でそのルートは動的レンダリングになるため、メモ一覧は常にリクエスト時取得です。ログイン中ユーザー固有のデータなのでこれで正しい。逆に「全員同じ・更新が稀」なデータ（公開記事一覧など）は revalidate でキャッシュした方が速くなります。
なお Next.js 15 以降、fetch はデフォルトでキャッシュされません（14 までとは逆）。キャッシュしたい時だけ明示的にオプトインします。

エラーはどこで拾う？

page.tsx では try/catch せず、失敗時は例外を投げっぱなしにしています。これは第9章で作る app/memos/error.tsx（セグメントのエラー境界）に自動で捕捉させる方針だからです。「正常系はページ、異常系は境界」と役割を分けると、ページ本体が読みやすく保てます。トップの page.tsx が try/catch しているのは、あえて「その場でエラー表示する」別方針の例です。

🛠 ハンズオン 2
  1. features/memo/memo.api.ts を作成する。
  2. app/memos/page.tsx を上記の取得版に更新する。
  3. API とスキーマを起動：pnpm --filter @repo/api-schema build → API も pnpm --filter api dev（別ターミナル）。
  4. /memos を開き、一覧が出ることを確認（データが無ければ「まだメモがありません」）。
  5. 実験：memo.api.ts の .parse(raw) を一時的に raw as GetMemoListResponse に変え、API を止めて壊れたレスポンスを返すと何が起きるか観察 → .parse に戻す。
Server Component でのデータ取得が、Client Component の useEffect+fetch より優れている点を2つ。
① ウォーターフォールとローディング状態が消える：サーバーで await して完成 HTML を返すので、クライアントで「読み込み中→データ」の二段階レンダリングが不要。② 秘密を隠せる：API トークンや API の URL、重いロジックがクライアント JS に載らない。加えて DB/API に物理的に近いサーバーで叩ける、バンドルが軽くなる、等。
このメモ一覧ページが「動的レンダリング」になる決め手は？
apiClient が内部で cookies() を呼んでいること。リクエスト固有の情報（cookie）に依存するルートは、Next.js が自動的に動的（リクエストごとに実行）と判定する。ビルド時に固定 HTML を作れないため。
