0. 全体像と土台を知る

コードを足す前に、すでに用意されている土台を把握します。ここを理解しておくと、以降の章で「なぜ自分でゼロから認証や fetch ラッパを書かないのか」が腑に落ちます。

この章のゴール
  • apps/web の現在の構成と、これから作る memo 機能の位置づけを理解する
  • ディレクトリ規約（app → components → features の依存方向）を頭に入れる
  • 開発サーバの起動と dev ログインができる

0.1 いま web にあるもの

現状の apps/web はユーザー情報を表示するサンプル1枚と、認証まわりの土台だけがあります。

0.2 土台①：認証付き API クライアント

web の鉄則は「ブラウザから Express API を直接叩かない」。必ずサーバー側（Server Component / Server Action / Route Handler）を経由します。その心臓が libs/api-client.ts です。

なぜブラウザから直接 API を叩かないのか

①認証トークンを HttpOnly cookie に隠せる — JS から読めない cookie に置くことで XSS でトークンを盗まれにくくします。ブラウザ fetch では Authorization ヘッダを JS が組み立てる必要があり、トークンが JS に露出します。
②401→refresh の自動リトライを1か所に集約できる。
③ API の origin をサーバー側だけで解決でき、CORS も不要。
あなたが memo 機能で書く memo.api.ts も、この apiClient を呼ぶだけにします。

0.3 土台②：型はすべて @repo/api-schema から

リクエスト/レスポンスの型は web で独自定義しません。共有パッケージ @repo/api-schema の Zod スキーマと z.infer 型を import します。memo の型はすでに定義済みです。

⚠ mobile 実装をそのまま真似しない

apps/mobile/src/features/memo/memo.api.ts は axios でブラウザから直接 API を叩き、コメントにも「型安全ではない」と書かれています。これはアンチパターンの実例として残っているものです。web ではこの章で学ぶベストプラクティス版（サーバ経由 + スキーマ検証）を作ります。「既存コードだから正しい」とは限らない、という視点も持ってください。

0.4 依存の方向：app → components → features

このプロジェクトはフロント3アプリ（web / admin / mobile）でディレクトリ規約を統一しています。ドメイン固有のロジックは機能単位（features/）で分け、依存の向きを一方向に保ちます。汎用フックや純関数などドメインを持たない共有コードだけを hooks/・libs/ の共有ティアに置きます（＝技術単位でドメインを散らかさない、という意味）。

ディレクトリ役割memo での例
app/memos/ルーティング＋ページ。薄く保つ（コンポーネントを組み合わせるだけ）page.tsx, [id]/page.tsx, actions.ts
components/features/memo/memo 固有の UI コンポーネントMemoList.tsx, MemoForm.tsx
components/ui/汎用 UI（Button, Input …）Button.tsx
features/memo/ロジックのみ（レンダリングなし）。API 通信・状態管理memo.api.ts, memo.state.ts

0.5 開発サーバを起動してログインする

なぜ dev ログインがあるのか / なぜ app/api/ 配下なのか

本番の Google OAuth を毎回踏むのは開発上つらいので、NODE_ENV !== "production" のときだけ有効な近道が用意されています。proxy.ts・Route Handler・API の3か所でガードされ、本番では 404 になります（第8章で詳解）。

配置方針：この dev ログインは route.ts（＝ HTTP エンドポイント）なので、route handler は例外なく app/api/* に置くという規約に従い app/api/dev/login/route.ts（URL は /api/dev/login）にあります。「dev 用だから」と app/dev/ のような例外を作らず、ページ（page.tsx）は app/*・エンドポイント（route.ts）は app/api/*、と置き場所を役割で一本化しておくと、後から迷いません。

🛠 ハンズオン 0
  1. pnpm --filter web dev で web を起動する。
  2. http://localhost:3000 を開く → 未ログインなら /sign-in にリダイレクトされることを確認。
  3. http://localhost:3000/api/dev/login?as=alice を踏み、トップでユーザー情報が表示されることを確認。
  4. apps/web/src/libs/api-client.ts を実際に開き、get/post/put/delete がどんな型を返すか眺めておく（第2章で使います）。
web では「ブラウザから Express API を直接 fetch する」のはなぜ禁止？ 主な理由を1つ挙げよ。
認証トークンを HttpOnly cookie に隠したまま、サーバー側でのみ Authorization ヘッダに載せられるから（XSS 対策）。加えて 401→refresh の集約、origin のサーバー解決、CORS 不要といった利点もある。トークンをブラウザ JS が扱うと露出リスクが上がる。
