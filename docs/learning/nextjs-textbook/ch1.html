1. App Router のファイル規約

App Router ではファイル名そのものが役割を持ちます。page.tsx を置けばルートになり、loading.tsx を隣に置くだけでローディング UI が有効になる。まずは空の /memos ページを作りながら、この「規約」を体で覚えます。

この章のゴール
  • app/ 配下の特殊ファイル（page / layout / loading / error / not-found）の役割を説明できる
  • ルートセグメントとルートグループ (...) の違いを理解する
  • 動く /memos ページ（静的な仮表示）と loading.tsx を作る

1.1 特殊ファイルの地図

App Router で意味を持つ主なファイルは次の通り。1つのフォルダ＝1つの URL セグメントで、その中に役割別のファイルを置きます。

ファイル役割境界(Server/Client)
page.tsxその URL の本体。これがある階層だけが公開ルートになる既定 Server
layout.tsx配下で共有される枠。再ナビゲーションでも再マウントされない既定 Server
loading.tsx配下の読み込み中に自動表示（内部的に <Suspense>）Server/Client 可
error.tsx配下で throw された実行時エラーを捕捉する境界Client 必須
not-found.tsxnotFound() 呼び出し・未定義 URL の表示Server 可
route.tsページではなく HTTP エンドポイント（GET/POST…）。page.tsx と同居不可Server
template.tsxlayout に似るが、遷移ごとに再マウントする枠Server/Client 可
なぜ「ファイル名が規約」なのが嬉しいのか

ルーティング設定やローディング/エラーの配線をコードで書かなくてよい。フォルダを見れば URL 構造と、どこにローディング・エラー処理があるかが一目で分かります。「設定より規約(convention over configuration)」の思想です。

1.2 まず動く空ページを置く

app/memos/page.tsx を作れば、それだけで /memos が生えます。まずはデータ取得なしの静的な仮表示から。

逐行のポイント

  • export default したコンポーネントがそのままページになる。関数名は自由（URL には無関係）。
  • "use client" を書いていないのでServer Component。サーバーでレンダリングされ、JS バンドルに載らない。
  • export const metadata で <title> などを宣言（詳細は第4章の Metadata API）。
  • web / admin の規約ではコンポーネントは function 宣言で書きます（API 側はアロー関数）。page.tsx は薄く保つのが原則。

1.3 loading.tsx を隣に置く

同じ階層に loading.tsx を置くと、この配下が「まだ準備中」の間 Next.js が自動でそれを表示します（次章でデータ取得を入れると効いてきます）。裏側では Next.js がページを <Suspense fallback={<Loading/>}> で包んでくれています。

1.4 ルートグループ (...) — URL を増やさずに整理する

フォルダ名を ( ) で囲むと、その階層はURL に現れません。共通レイアウトを与えたいページ群をまとめるのに使います。例えばログイン後の画面をまとめた (dashboard) グループ。

app/ ├─ (dashboard)/ # ← URL には出ない。ダッシュボード共通レイアウト用 │ ├─ layout.tsx # サイドバー等の共通枠 │ ├─ memos/page.tsx # → /memos │ └─ settings/page.tsx # → /settings └─ sign-in/page.tsx # → /sign-in（レイアウトを共有しない）
本書での扱い

本書では話を単純にするため app/memos/ 直下に作ります。実務でヘッダ付きの共通レイアウトを与えたくなったら app/(dashboard)/memos/ に移し、(dashboard)/layout.tsx を足す、という発展を第3章のコラムで触れます。CLAUDE.md でも Server Action の配置例として app/(dashboard)/categories/actions.ts が示されています。

🔧 互換メモ：params / searchParams は Promise

Next.js 15 以降、page.tsx が受け取る params と searchParams は Promise になりました（await が必要）。既存の sign-in/page.tsx でも const { error } = await searchParams と書かれています。第4・5章で実際に使います。

🛠 ハンズオン 1
  1. apps/web/src/app/memos/page.tsx を上記の内容で新規作成する。
  2. /memos にアクセスし、見出しが表示されることを確認（未ログインなら先に /api/dev/login?as=alice）。
  3. loading.tsx も作成する。今はまだデータ取得が無いので一瞬しか出ないが、第2章で効いてくる。
  4. 試しに page.tsx の関数名を Foo に変えても /memos が動くことを確認 → 関数名は URL に無関係だと体感する。
layout.tsx と template.tsx の一番の違いは？
再マウントされるか否か。layout はページ間遷移でも状態を保持したまま再マウントされない（共有の枠として残る）。template は遷移ごとに新しいインスタンスとして再マウントされるので、遷移のたびに入場アニメを走らせたい／useEffect を毎回動かしたい、といった時に使う。
error.tsx だけが "use client" 必須なのはなぜ？
エラー境界は React の Error Boundary の仕組みを使い、reset() で再試行するためにクライアント側の状態とイベントを扱う必要があるから。Server Component は状態やイベントを持てないため、境界はクライアントで動く必要がある。
