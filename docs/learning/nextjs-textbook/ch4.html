4. 動的ルートと詳細ページ

一覧の各項目は /memos/1, /memos/2 … と ID で開けるようにします。URL の一部を変数として受け取る動的ルート [id] を作りながら、await params・notFound()・generateMetadata を学びます。

この章のゴール
  • [id] 動的セグメントと await params を使える
  • 存在しない ID で notFound() → not-found.tsx に繋げられる
  • generateMetadata でページごとの <title> を動的生成できる
  • ルートパラメータも Zod で検証する（プロジェクト規約）

4.2 [id] ページを作る

フォルダ名を [id] にすると、その部分の URL が params.id として渡ります。Next.js 15 以降、params は Promise なので await します。

逐行のポイント

  • getMemoPathParamSchema は z.coerce.number().int().positive()。文字列→数値変換と検証を Zod に任せる（Number() や parseInt を手書きしない、が規約）。
  • 不正 ID（/memos/abc）は safeParse 失敗 → notFound()。存在しない ID は API が 404 → ApiClientError.status === 404 を見て notFound()。
  • notFound() は最寄りの not-found.tsx を表示する。既存の app/not-found.tsx がそのまま使われる（第9章でメモ専用に上書きも可）。
  • 404 以外の失敗（500 等）は throw して error.tsx に任せる。「無い」と「壊れた」を区別するのが ApiClientError が status を保持している狙い（第9章）。
なぜ generateMetadata をわざわざ書くのか

静的な export const metadata ではページ内容に応じた <title> を作れません。generateMetadata は params を受け取れる非同期関数なので、「メモのタイトルをそのままページタイトルに」といった動的な SEO / タブ表示ができます。Next.js は本文レンダリングと generateMetadata の getDetail を重複取得しないよう自動でメモ化（dedupe）してくれるので、2回書いても API は1回で済みます。

4.3 SSG / ISR はこのページに使う？

generateStaticParams を書くと、ビルド時に [id] の一覧を先に生成して静的化（SSG）できます。ブログ記事のような「全員に同じ・更新が稀」なコンテンツでは強力です。

メモは動的でよい

メモはログインユーザー固有・頻繁に増減し、apiClient が cookies() を使う（＝動的）ため、SSG/ISR は不向きです。「認証が要る・ユーザーごとに違う・すぐ変わる」データは動的レンダリングが正解。逆に公開・共通・低頻度更新なら ISR を検討する、という判断軸を持っておきましょう。

🔧 互換メモ：動的 API はすべて Promise

Next.js 15/16 では params / searchParams / cookies() / headers() がすべて非同期（Promise）です。同期的に params.id と書くと型エラー・実行時警告になります。必ず await しましょう。

  5. ブラウザのタブにメモのタイトルが出ることを確認（generateMetadata の効果）。
「不正な ID（abc）」と「存在しない ID（999999）」を、コードではどう区別して両方 404 にしている？
不正 ID は getMemoPathParamSchema.safeParse が失敗する（数値でない/正の整数でない）ので、その場で notFound()。存在しない ID は形式は正しいので API まで行き、API が返す 404 を ApiClientError.status で判定して notFound()。500 など他の失敗は throw して error.tsx に回す。
