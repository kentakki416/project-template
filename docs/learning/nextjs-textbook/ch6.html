6. 編集・削除と楽観的更新

作成ができたら、更新（PUT）と削除（DELETE）を足します。ここで フォームを再利用する設計、引数付き Server Action（.bind）、そして楽観的更新（useOptimistic）で「押した瞬間に反映」する UX を学びます。

この章のゴール
  • 作成フォームを汎用化して編集で再利用できる
  • .bind() で ID 付きの Server Action を作れる
  • useOptimistic で削除を即時反映（見かけ上）できる

6.1 取得層に update / remove を足す

6.2 フォームを再利用できるようにする

第5章の MemoForm は createMemoAction に直結していました。作成と編集で見た目は同じなので、action と初期値を props で受け取る汎用フォームにリファクタします。まず状態型を共通化します。

new/page.tsx の呼び出しも更新

汎用化に伴い、作成ページは <MemoForm action={createMemoAction} submitLabel="作成する" /> のように action を渡す形に変わります（createMemoAction を import して渡す）。Server Action を Client Component に props で渡せるのが便利なところ。

6.3 編集ページと .bind()

更新には「どのメモか」の id が要ります。updateMemoAction は第1引数が id なので、.bind(null, memo.id) で id を固定した新しい関数を作り、それを MemoForm に渡します。すると MemoForm 側は (prev, formData) の普通の action として扱えます。

6.4 削除ボタン

削除はフォームの action に deleteMemoAction.bind(null, memo.id) を渡すだけ。詳細ページに置きます。確認ダイアログはクライアント挙動なので小さな Client 島にします。

詳細ページ（第4章）にボタンを差し込みます。

6.5 発展：useOptimistic で「押した瞬間」に消す

上の削除は「サーバー処理 → revalidate → 再描画」なので、一覧から消えるまで一瞬待ちます。楽観的更新は「成功する前提で先に UI を更新」し、体感を速くします。一覧をクライアント側でラップして useOptimistic を使う例です（発展・任意）。

楽観的更新はいつ使う？

「ほぼ確実に成功し、失敗してもリカバリが軽い」操作（いいね、削除、並べ替え）に向きます。useOptimistic の良い点は、Server Action が失敗したら自動で楽観状態が破棄され、本物のサーバー状態に戻ること。逆に「失敗が致命的・整合性が命」の操作（決済など）は、確定を待ってから反映する方が安全です。

updateMemoAction.bind(null, memo.id) の .bind は何をしている？
関数の第1引数（ここでは id）を memo.id に固定した新しい関数を作っている（部分適用）。結果として (prev, formData) => ... という useActionState が期待する形の action になり、id はクロージャで保持される。第1引数 null は this の指定（Server Action では使わないので null）。
useOptimistic で削除して、もし Server Action が失敗したら画面はどうなる？
楽観的に消していた項目が自動的に元に戻る（本物のサーバー状態＝まだ存在するリストに復帰する）。useOptimistic は「保留中の楽観状態」を一時的に重ねているだけなので、アクション完了（成功でも失敗でも）後は確定したサーバー状態に収束する。
